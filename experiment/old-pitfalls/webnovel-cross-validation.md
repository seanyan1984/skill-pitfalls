## 9. 已踩过的坑

1. **cron tick lock 阻塞（Issue #3752，已知未修复）**：长任务在 `tick()` 内持有 `.tick.lock`，阻塞后续所有 tick。这不是 ticker 坏了，是 known issue——一个慢 job 会独占整个 scheduler 直到跑完。**应对**：① 链式 cron 是最佳方案——天然串行，不存在并行冲突；② `action='run'` 可强制触发下一个 job；③ 官方故障排除文档：`hermes-agent.nousresearch.com/docs/guides/cron-troubleshooting`。
2. **delegate_task 超时**：5本书 ~600s 必超时。**推荐 1 本书/subagent**（最稳）。2 本/次有成功案例（金手指维度 ~265s）但也有全部超时案例（人设维度批3/4，600s 双双超时且未写出文件）。取决于章节长度和内容密度——无法预测哪批会爆，所以 1本/subagent 是唯一可靠默认。并行超时后不要重试同配置，直接拆成 1本/subagent 重跑。
3. **共享文件写入冲突**：多个 agent 写同一文件会互相覆盖。已改为独立文件。
17. **汇总分两级，执行方式不同**：
    - **批摘要（读取 3 个 Agent 文件 ~12KB → 交叉验证 → 写 1 份汇总）**：用 delegate_task batch 2 并发，每轮 2 个批摘要同时跑。实测 125-180 秒/批，稳定不超时。5 批汇总分 3 轮完成（2+2+1）。
    - **总报告（读取 5 份汇总 ~65KB → 合成全景图谱+模式归纳+质量评估）**：**必须在主会话中完成，不要 delegate**。实测 delegate 两次均 600s 超时（子 agent 要读 5 个大文件 + 合成长报告）。正确做法：① 主会话 `read_file` 逐份读取 5 份汇总；② 直接 `write_file` 写总报告。
21. **分析密度因维度而异**：不同维度的分析密度不同，影响耗时。金手指维度逐章找金手指使用场景（稀疏），人设维度逐章抠角色细节（密集）。所以金手指 2本/subagent 能跑完，人设不一定。不能跨维度套用耗时经验。
22. **预提取正文到 raw/*.txt 是必须步骤**：不要让子 agent 直接查 DB——20+ 次 SQL 查询耗尽 600s 超时。预提取后只需 2-3 次 read_file，140-190s 稳完。在跑 delegate_task 前先执行正文预提取。
